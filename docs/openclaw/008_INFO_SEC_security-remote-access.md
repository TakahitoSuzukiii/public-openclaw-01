# OpenClaw のリモートアクセスをセキュアにする選択肢

OpenClaw にリモートからアクセスする際の代表的なセキュア化手段について、「本当にセキュアになるのか」を公式ドキュメントに照らして整理する。対象は以下:

- リモートアクセスが必要な場合: **SSH トンネル** または **VPN（Tailscale / WireGuard）**
- **リバースプロキシ（Nginx / Caddy）で認証を追加**

> 本書は一般的な解説。特定の環境の設定値・構成には触れない。

---

## 0. 結論（先に要約）

- これらは **「リモートアクセスを許可するとき」に効く対策**。逆に言えば、**リモートアクセスが不要なら、そもそも外部に公開しないのが最も安全**。
- リモートで使いたい場合は、**「公開せずトンネル/VPN で繋ぐ」が最優先**。
- リバースプロキシ＋認証は、複数人やブラウザ常時公開など、**より高度なケース向け**。個人・単独利用ではオーバースペックになりがち。

---

## 1. 前提：OpenClaw のネットワーク設計（一般）

OpenClaw 公式ドキュメントは **「Loopback first（まずループバック）」** を基本方針としている。

> **用語: loopback（ループバック）** = 自分のPC内部だけで完結する特別なアドレス `127.0.0.1`。外のネットワークには一切出ない。「玄関のドアを内側からしか開けられない」イメージ。

- ゲートウェイの WebSocket は既定でループバックにバインドされる設計。
- ループバック以外にバインドする場合は、適切な認証（トークン / パスワード / trusted-proxy）が必須。
- 接続認証のモードは `none` / `token` / `password` / `trusted-proxy` が用意されている。

つまり「外から入ってくる口（インバウンド）を開かない限り、外部からは到達できない」のが基本。リモートアクセスを足すときだけ、以下の対策を検討する。

---

## 2. なぜ「リモートアクセス時」に対策が要るのか

ループバックをやめて外から繋げるようにする＝**インターネットに口を開ける**こと。何もしないと、世界中から接続試行・パスワード総当たり・脆弱性スキャンの対象になる。

そこで「**公開せずに、自分だけが安全に繋ぐ**」ための仕組みが必要になる。それが SSH トンネル / VPN / リバースプロキシ。

---

## 3. 各手段の用語解説と評価（具体例つき）

### 3-1. SSH トンネル（ポートフォワーディング）

- **用語**: すでにある SSH 接続の「中」に、別の通信を通す仕組み。OpenClaw のループバックポートを、手元のPCのループバックに“引き込む”。
- **具体例**: 手元のPCで次を実行すると、手元の `127.0.0.1:<port>` がサーバ側の OpenClaw に繋がる。
  ```bash
  ssh -N -L <port>:127.0.0.1:<port> <your-user>@<gateway-host>
  ```
  以後 `openclaw health` などがトンネル経由でリモートに届く。
- **評価**: ✅ **正しい・安全**。サーバ側はループバックのまま（公開しない）。SSH の認証（鍵）をそのまま流用できる。**追加インストール不要**で最小構成。必要なときだけ繋ぐ「オンデマンド」用途に最適。
- **短所**: 繋ぐたびにトンネルを張る手間。常時アクセスにはやや不便。

### 3-2. VPN：Tailscale / WireGuard

- **用語: VPN** = 複数の機器を、あたかも同じ安全な専用ネットワークにいるように繋ぐトンネル。
- **用語: WireGuard** = 軽量・高速な VPN の「プロトコル（通信方式）」。設定がシンプルで高速なのが特徴。
- **用語: Tailscale** = その WireGuard を土台にした VPN サービス。面倒な鍵交換やファイアウォール設定を自動化し、機器同士を「メッシュ（網の目）」で直結する。
- **具体例**: サーバと自分のスマホ・PC を Tailscale に参加させると、各機器に `100.x.y.z` のような**プライベートアドレス**が付く。そのアドレスでだけ OpenClaw に届く（外部からは見えない）。
- **OpenClaw との相性**: 公式は **Tailscale Serve** を統合サポート。**ゲートウェイはループバックのまま**、Tailscale 側が HTTPS と本人確認ヘッダを付けてくれる。設定例:
  ```jsonc
  // openclaw.json
  "gateway": {
    "bind": "loopback",
    "tailscale": { "mode": "serve" },     // tailnet 限定で公開
    "auth": { "mode": "token", "allowTailscale": true }
  }
  ```
- **評価**: ✅ **正しい・推奨**。「常時アクセスしたい」「スマホからも使いたい」場合のベスト。SSH トンネルより UX が良い。Tailscale は導入が容易、WireGuard 直は自前管理で更に最小だが手間。
- **注意**: `funnel`（公開モード）は文字通りインターネット公開になるため、共有パスワード必須＆基本は使わない。

### 3-3. リバースプロキシ（Nginx / Caddy）＋ 認証

- **用語: リバースプロキシ** = 利用者と OpenClaw の「間」に立つ受付係。外からのアクセスをまず受け、TLS（暗号化）終端や認証を済ませてから、内側の OpenClaw に転送する。
- **用語: Nginx / Caddy** = 代表的なリバースプロキシソフト。Caddy は HTTPS 証明書を自動取得できて手軽。
- **用語: 認証の追加（identity-aware proxy）** = プロキシ段で「ログイン」を必須にする。例: Google / GitHub の OAuth ログインを通さないと OpenClaw に到達できないようにする。
- **具体例**: `https://openclaw.example.com` にアクセス → Caddy が Google ログインを要求 → 認証成功者だけ内部のループバックに転送。
- **OpenClaw との連携**: `gateway.auth.mode: "trusted-proxy"` で、認証をプロキシに委譲できる（プロキシが付けた `x-forwarded-user` 等のヘッダで本人を識別）。
  ```jsonc
  "gateway": {
    "auth": { "mode": "trusted-proxy" },
    "trustedProxies": ["<proxy-ip>"]      // プロキシ以外からの直結は拒否
  }
  ```
- **評価**: ⚠️ **正しいが、用途を選ぶ**。公式の指針:
  - **向いている**: 複数ユーザー、ブラウザで常時公開、Kubernetes/コンテナでプロキシが唯一の経路、OAuth で組織的にアクセス制御したい。
  - **向いていない（オーバースペック）**: **個人・単独利用なら不要**。公式も「単独利用なら Tailscale Serve + loopback の方が簡単」と明言。
  - **危険な誤設定**: プロキシを迂回してゲートウェイに届く抜け道（ファイアウォール穴・内部ネットワーク）が1つでもあると、認証を素通りされる。`trusted-proxy` は誤設定が即・全公開に繋がるため慎重に。

## 4. シナリオ別おすすめ

| 状況 | 推奨 | 理由 |
|---|---|---|
| リモート不要（公開しない運用） | **何もしない（loopback 維持）** | 外部非公開が最も安全 |
| たまに手元PCから操作したい | **SSH トンネル** | 追加導入ゼロ・公開しない |
| スマホ含め常時アクセスしたい | **Tailscale（Serve）** | loopback 維持のまま安全・UX 良好 |
| 複数人・ブラウザ常時公開・組織運用 | **リバースプロキシ＋OAuth（trusted-proxy）** | 本人確認と TLS を集中管理 |

---

## 5. 推奨（一般）

1. **リモートが不要なら、公開しない（loopback のまま）のが最も安全。**
2. リモートが必要になったら、まず **SSH トンネル**（最小・即可能）。
3. 常用するなら **Tailscale Serve**（loopback 維持で安全・楽）。
4. リバースプロキシ＋認証は、**複数人運用や常時公開**になって初めて検討。個人利用では非推奨（複雑さとリスクが増える割に利点が薄い）。

> 要は「**公開しないのが最強。どうしても繋ぐなら、公開せずトンネル/VPN で**」。リバプロ＋認証は規模が大きくなった時の選択肢。
